# The Mock KMS

The Mock KMS is the key management layer of the simulator. It is a lightweight HTTP server that maintains a pool of key units, generates them at a fixed rate, and provisions them to the Agent on request. It deliberately models only the behaviors relevant to the simulator — key pool management and link provisioning — without implementing any of the quantum hardware or cryptographic material of a real KMS.

## What It Mocks

In a real QKDN, the KMS sits on top of quantum hardware. Keys are generated by the BB84 protocol running on optical fiber, pushed into the pool after sifting, error correction, and privacy amplification, and assigned key IDs for tracking. The pool refills continuously as the quantum link generates new key material.

This simulator has none of that hardware. Instead, the KMS runs a background coroutine that adds one key unit to the pool every 100ms, up to a maximum of 100 units:

```python
async def _generate_keys(self):
    while True:
        async with self._lock:
            if self._eskr_pool < self.MAX_ESKR_POOL:
                self._eskr_pool += 1
        await asyncio.sleep(0.1)
```

This approximates the fixed, continuous generation rate of a real QKD link. The pool ceiling (`MAX_ESKR_POOL = 100`) approximates the finite capacity of a real key store.

What is **not** mocked: actual key material. The pool tracks quantities only — there are no cryptographic keys generated or stored. In a real KMS, a race condition on the pool would not just cause accounting errors; it could result in two different secure communications receiving the same key material, completely breaking confidentiality. The lock here enforces the same invariant, even without real keys.

## The ESKR Pool

**ESKR** stands for Effective Secret Key Rate — the rate at which usable key material is available on a link. In a real system, ESKR accounts for losses due to channel noise, error correction overhead, and privacy amplification. Here it is simplified to a single integer representing available key units.

The pool has three properties relevant to provisioning:

- It has a **maximum capacity** (100 units)
- It **refills at a fixed rate** (1 unit per 100ms = 10 units/second)
- It **depletes when links are provisioned** — each provisioned link consumes `key_rate_required` units permanently

There is no concept of link teardown in this simulator — once ESKR is consumed, it is gone until the pool refills naturally.

## The Lock and Why It Matters

The KMS uses an `asyncio.Lock` on every operation that reads or writes the pool. This is not optional — it is a correctness requirement.

The Agent and KMS run as concurrent async applications. When multiple provisioning requests arrive in quick succession, `asyncio` can interleave their coroutines at any `await` point. Without a lock, two requests could both pass the pool availability check before either has deducted from it:

```
Request A: pool = 15, needs 10 → check passes
Request B: pool = 15, needs 10 → check passes   <-- interleaved before A deducts
Request A: pool -= 10 → pool = 5
Request B: pool -= 10 → pool = -5               <-- race condition
```

The lock ensures the check and the deduction happen atomically — no other coroutine can interleave between them.

## Endpoints

The KMS exposes three endpoints, all intended for internal use by the Agent only. Clients never call the KMS directly.

### `GET /api/status`

Returns the current state of the KMS:

```json
{
  "status": "online",
  "eskr_available": 73,
  "active_links": 2
}
```

This is called by the Agent's background polling task every 2 seconds to keep its local state up to date.

### `GET /api/capabilities`

Returns static information about this KMS node:

```json
{
  "node_id": "node-1",
  "max_eskr_pool": 100,
  "supported_sla_levels": ["critical", "high", "normal"]
}
```

### `POST /api/link_config`

The provisioning endpoint. Accepts:

```json
{
  "link_id": "link-a3f2b1c4",
  "target_node": "node-2",
  "sla_level": "high",
  "key_rate_required": 20
}
```

All four fields are required. `sla_level` must be one of `critical`, `high`, `normal`. The KMS validates the payload, acquires the lock, checks pool availability, and returns either:

```json
{ "status": "success", "link_id": "link-a3f2b1c4", "eskr_consumed": 20 }
```

or:

```json
{ "status": "failed", "link_id": "link-a3f2b1c4", "eskr_consumed": 0, "reason": "insufficient_eskr" }
```

### `GET /health`

Returns the health status of the KMS, including current pool level and active link count. Used for operational monitoring.

## What a Real KMS Would Add

For context, a production KMS — such as those deployed in real QKDN testbeds — would additionally handle:

- Actual cryptographic key material with unique key IDs (per ETSI GS QKD 014)
- Key expiry and rotation
- Authentication of callers (only pre-authorized agents may request keys)
- Multiple simultaneous QKD links, each with their own pool
- Hardware Security Module (HSM) integration for secure key storage

The simulator deliberately omits these to keep the focus on the network coordination logic — which is where the interesting engineering lives in the SDN Agent.
